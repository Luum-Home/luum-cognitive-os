# ADR-285: Skill Registry Runtime Drift Detection

## Status

Accepted and implemented — 2026-05-13.

<!-- SCOPE: OS -->

**Date**: 2026-05-13

## Context

`skills/REGISTRY.lock` records the SHA-256 hash of every skill file at
promotion time. It contains 166 entries and is regenerated by the promotion
proposer. At present the file is consumed only by:

- `scripts/cos_promotion_proposer.py` — during cross-instance evidence builds
- `lib/cross_instance_learning.py` — during Shape-B federation readiness checks
- `tests/contracts/` — static contract tests

No runtime enforcement exists. A skill file can be silently edited after
promotion — accidentally or by an agent with write access — and the OS will
continue loading the mutated version without any awareness of the discrepancy.
This means `REGISTRY.lock` functions as documentation rather than a security
control: the lockfile proves integrity *at the moment of promotion*, not at
the moment of execution.

The gap matters for two reasons:

1. **Trust surface.** Skills are loaded and invoked by agents. A mutated skill
   could introduce hallucinated criteria, weakened safety rules, or changed
   tool invocations. Without runtime verification, mutation is invisible until
   a human audits the diff manually.

2. **Federation correctness.** Shape-B cross-instance evidence exchange assumes
   the sending instance presents skills that match their locked hashes. Drift on
   the sending side invalidates the evidence bundle silently.

A lockfile without runtime enforcement is only documentation.

## Decision

Add a `SkillDriftDetector` that runs at `SessionStart` and compares the on-disk
SHA-256 of each locked skill against the hash recorded in `REGISTRY.lock`.

### Policy

The detector supports two modes controlled by the `COS_SKILL_DRIFT_POLICY`
environment variable:

| Value | Behaviour |
|---|---|
| `warn` (default) | Log drifted skills to stderr and to the audit trail. Never block. |
| `block` | Exit non-zero when drift is detected, preventing session start. Reserved for strict or production-locked deployments. |

Default is `warn` to avoid operational disruption. Operators who require
tamper-evidence can set `COS_SKILL_DRIFT_POLICY=block` in their environment.

### Performance budget

The detector must complete in under 50 ms for a full registry of 175 skills.
Hashing every file on every session start is expensive. A mtime-keyed cache at
`.cognitive-os/state/skill-hash-cache.json` avoids re-hashing files whose
modification time has not changed since the last check:

```
cache key  = relative skill path
cache value = { "mtime": <float>, "sha256": <hex string> }
```

A cache hit is valid when `os.stat(path).st_mtime == cache[key]["mtime"]`.
On cache miss the file is re-hashed and the cache entry is updated atomically.

### Audit trail

Every drift event is appended to `.cognitive-os/metrics/skill-drift.jsonl` in
NDJSON format:

```json
{"ts": "2026-05-13T10:00:00+00:00", "skill": "skills/add-hook/SKILL.md", "expected": "abc123", "actual": "def456", "policy": "warn"}
```

This provides a durable record for post-incident analysis and for the session
changelog to reference.

## Consequences

- **Positive**: Mutations to locked skills are surfaced at session start before
  any agent tool invocations execute. Audit trail supports incident forensics.
- **Positive**: mtime cache means the steady-state cost for an unmodified tree
  is a stat-only pass (~1–2 µs per file), well within the 50 ms budget.
- **Positive**: `warn` default means the hook is safe to deploy immediately
  without risk of blocking legitimate sessions.
- **Negative**: The cache introduces a small window where a mutation that
  preserves mtime (e.g. `touch -t`) would be missed until the cache is
  invalidated. This is an acceptable trade-off for the performance budget; a
  strict deployment should set `COS_SKILL_DRIFT_POLICY=block` and accept the
  full-hash cost on each session.
- **Negative**: The hook adds a few milliseconds of Python startup to
  SessionStart. The killswitch (`COS_DISABLE_SKILL_DRIFT_DETECTOR=1`) allows
  operators to opt out.

## Alternatives rejected

1. **Pre-commit hook only.** Pre-commit verification does not protect against
   post-commit mutations inside a running session or across sessions.
2. **Periodic cron.** A periodic check would miss drift that occurs and is
   reverted between checks. SessionStart catches drift before it can be acted
   on by any tool in that session.
3. **Block by default.** Rejected for initial rollout because it could prevent
   legitimate sessions if the lock is stale due to intentional promotion work
   in progress. Operators who need strict enforcement can set the block policy.
