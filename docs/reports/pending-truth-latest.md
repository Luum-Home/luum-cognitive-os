feature-reality-matrix). The surfaces remain authoritative for their own narrative

couldn't answer consistently: **'how many actionable pending items exist?'**

Practical pattern:
1. At session start → run `scripts/cos-pending-truth-aggregator --write` to refresh
2. To find next work → grep this file for items with low `last_verified` age
3. To know if item X is really still pending → check `status` (or run Slice B verifier
   when it lands)
4. When code lands that closes a plan item → until Slice C lands, manually update the
   source plan checkbox `[ ]` → `[x] (verified: commit-SHA path)`

**When surface and code disagree, code wins** (ADR-244 trust-score precedent).
If a plan checkbox says `[ ]` but ledger says `verified-done`, the plan is stale.
